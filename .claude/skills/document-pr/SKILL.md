name: document-pr
description: "Generate a Markdown record of the work in the current PR/branch and store it under docs/ as <timestamp>-<slug>.md, then prepend an entry to the docs/README.md index so the newest PR is always at the top. Use after opening a PR (or when asked to 'document this PR/work')."
---

### Purpose

Produce a durable, human-readable record of what a PR changed and why, stored in
the repo so the team has a chronological log of shipped work — newest first.

### When to run

- Right after creating a PR (`gh pr create`), as the final step of the flow.
- When the user asks to "document this PR", "write a PR doc", or "record this work".

### Inputs to gather (in this order)

1. **Branch & PR metadata**
   - `git rev-parse --abbrev-ref HEAD` — current branch.
   - `gh pr view --json number,title,body,url,headRefName,baseRefName` — if a PR
     exists for the branch. If no PR yet, fall back to the branch name + the
     latest commit subjects and note "PR not yet opened" in the doc.
2. **The diff being summarized**
   - `git log --oneline <base>..HEAD` (base is the PR's baseRefName, else `main`).
   - `git diff --stat <base>...HEAD` for the file-level shape.
   - Read the actual diff for anything non-obvious; do NOT invent changes.

### Output 1 — the per-PR doc

Path: `docs/<timestamp>-<slug>.md`

- `<timestamp>`: `date +%Y-%m-%dT%H%M` (local time, minute precision, sortable).
- `<slug>`: kebab-case of the PR title (or branch name), lowercased, ~6 words max,
  strip conventional-commit prefixes like `feat:`/`refactor:`.
- Example: `docs/2026-07-08T1430-migrate-graphql-to-rest.md`.

Template:

```markdown
# <PR title>

- **PR:** #<number> — <url>   (or "not yet opened")
- **Branch:** <headRef> → <baseRef>
- **Date:** <YYYY-MM-DD HH:MM>
- **Author:** <git user.name>

## Summary
<2–4 sentences: what changed and why. Plain language.>

## Changes
<Bulleted list grouped by area/package. One line each, most significant first.>

## Verification
<How it was validated: analyze/tests/manual run. State outcomes plainly.>

## Notes / follow-ups
<Out-of-scope items, known gaps, linked issues. Omit if none.>
```

Keep it factual and concise. Report outcomes honestly (if tests failed or a step
was skipped, say so).

### Output 2 — the index (newest at top)

Path: `docs/README.md`

- If it does not exist, create it with a heading and an empty list:
  ```markdown
  # PR / Work Log

  Newest first. Generated by the `document-pr` skill.

  <!-- entries below -->
  ```
- **Prepend** a one-line entry immediately under the `<!-- entries below -->`
  marker so the newest is always at the top:
  ```markdown
  - **<YYYY-MM-DD>** · [<PR title>](<timestamp>-<slug>.md) — <one-line hook>
  ```
- Never rewrite existing entries; only insert the new one at the top.

### Steps

1. Gather the inputs above.
2. Create `docs/` and `docs/README.md` if missing.
3. Write `docs/<timestamp>-<slug>.md` from the template.
4. Prepend the index entry to `docs/README.md`.
5. Tell the user the doc path and the index line you added. Do not commit or push
   unless the user asks.
